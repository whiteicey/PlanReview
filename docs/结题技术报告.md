## 摘要

本项目完成了一套面向油气田**开发方案 DOCX** 的**本地优先智能初审内核**：将方案解析为带原文定位（SourceSpan）的参数事实，经**三值规则引擎**做完整性/一致性/汇总/产能/版本/术语/证据检查，叠加**可选 LLM 二审**（默认 Mock，可选 Anthropic 兼容网关），输出可追溯的问题清单，并支持专家复核与 Excel/Word/匿名包导出。

**定位恒定：** 智能初审 + 专家复核，**不是**自动审批，**不是**正式审查结论签发。

**核心技术主张：**

1. **确定性优先** — 可编码的规则用纯函数 operator 判定，三值逻辑 fail-closed；  
2. **第一性原理** — Finding 必须由事实 + 证据推导，禁止用正文结论句「猜答案」；  
3. **可追溯** — 每个问题绑定原文 span，导出可读位置；  
4. **诚实** — 证据不足报 UNKNOWN/转人工；未跑项与能力边界写入文档，不虚构指标。

**当前验收快照（2026-07-15，无外部 DEMO 环境变量）：**  
`python -m pytest -q` → **337 passed, 8 skipped**。  
（设置 `REVIEW_DEMO_ROOT` 指向示例包后可激活 golden 回归；本报告不虚构 golden 全量数字，以当次执行输出为准。）

### 1.1 业务痛点（与开题一致）

| 特征 | 现状（开题口径） |
|------|------------------|

| 单案轮次 | 约 3–4 轮 |

| 目标 | 本地版实现情况 |
|------|----------------|
| 方案解析与完整性检查 | ✅ DOCX → SourceSpan；章节/参数表/回复表规则 |

- 自动审批 / 自动签发正式结论  
- 自动修改源 DOCX  
- PDF 文本层 / OCR / 扫描件（接口预留，用户未要求前不实现）  
- 公网多租户 SaaS  
- 无证据的优劣评价

## 二、总体技术架构

### 2.1 架构原则

```
大模型 + 规则引擎 +（可扩展）行业知识/术语
        ↑
   规则保确定性；LLM 做独立二审；知识以 YAML 术语/规则声明注入

与开题「混合架构」一致，本地版将**知识库**首期落实为：

- 外部权威规则包（DEMO ruleset YAML）  
- 术语别名表（terminology YAML）  
- 仓库自有规则（`repo_rules.yaml`）  

向量检索 / 规范全文 RAG **明确延后**。

### 3.3 规则清单（12 条，权威以代码/YAML 为准）

运行时加载结果 = **外部权威 10 条** + **仓库自有 2 条** = **12 条**。  
Operator 白名单共 12 个纯函数（与规则条数一一对应到具体 operator，见下表）。

#### 3.3.1 外部权威规则（10）— `本地版示例数据包/rules/ruleset-demo-0.1.yaml`

| 规则 ID | 名称 | 类别 | 严重度 | Operator | 判定意图（业务话） | on_missing |
|---------|------|------|--------|----------|-------------------|------------|
| **COMPLETENESS-001** | 必备章节存在 | completeness | medium | `required_sections_exist` | 方案是否具备总则、概况、开发部署、产能预测、井工程、地面工程、结论等必备章节 | fail |
| **COMPLETENESS-002** | 关键参数表存在 | completeness | medium | `required_parameter_table_exists` | 是否存在含「关键参数/主要技术指标/井数」等关键词的参数表区域 | unknown |
| **CONSISTENCY-001** | 开发井总数跨位置一致 | consistency | high | `all_equal` | 摘要/部署/参数表等处「开发井总数」在同口径下是否同一数值 | unknown |
| **CONSISTENCY-002** | 分类井数等于总井数 | aggregation | high | `sum_equals` | 生产井数 + 评价/探井数 是否等于 开发井总数 | unknown |
| **CONSISTENCY-003** | 单井产能与总设计产能关系 | aggregation | medium | `product_approximately_equals` | 开发井总数 × 单井设计产能 ≈ 总设计产能（默认相对容差 5%） | unknown |
| **CAPACITY-001** | 高峰产量不超过地面处理能力 | cross_domain | high | `less_or_equal` | 高峰产量是否 ≤ 地面处理能力（跨专业容量校核） | unknown |
| **VERSION-001** | 关键参数变化需要说明 | version_change | medium | `change_requires_reason` | 多版本间关键参数变化时，意见回复区是否出现原因类表述；单版本无跨版本变更则 PASS | suspected→unknown+人工 |
| **VERSION-002** | 上一轮意见应存在回复状态 | version_change | medium | `issue_response_status_exists` | 审查意见回复表是否存在非空状态单元格 | unknown |
| **TERM-001** | 参数别名应归一 | terminology | low | `alias_normalization` | 参数别名是否应按术语表归一（**结题 review 指出：生产路径存在假绿风险，见 code-review R1/R2**） | unknown |
| **EVIDENCE-001** | 问题必须关联原文证据 | evidence | high | `evidence_required` | 证据充分性门禁（**结题 review 指出：当前用全文 span 数，真实文档易恒 PASS，见 R3**） | block |

必备章节列表（COMPLETENESS-001 配置原文）：  
`总则与编制说明`、`项目概况`、`开发部署方案`、`产能与生产预测`、`井工程方案`、`地面工程方案`、`结论与实施建议`。

VERSION-001 关注的参数族（YAML `parameters`）：  
`开发井总数`、`单井设计产能`、`总设计产能`、`高峰产量`、`地面处理能力`、`建设周期`、`首次投产时间`。

#### 3.3.2 仓库自有规则（2）— `app/rules/repo_rules.yaml`（版本标记 `0.1.0-repo`）

| 规则 ID | 名称 | 类别 | 严重度 | Operator | 判定意图 | 说明 |
|---------|------|------|--------|----------|----------|------|
| **COMPLETENESS-003** | 审查意见回复表状态完整 | completeness | medium | `reply_table_status_complete` | 回复表每一数据行是否都有状态格（表头用「意见编号/意见」+「回复/状态」识别列） | 外部 10 条未覆盖「行级漏填」 |
| **TERM-002** | 正文参数别名未归一 | terminology | low | `prose_alias_unnormalized` | 正文叙事中是否出现与规范名不同的别名（无单位、抽不成 fact 时仍可抓） | 加载时注入**全量**术语 aliases |

#### 3.3.3 Operator 白名单（12，与 `OPERATOR_NAMES` 一致）

```
required_sections_exist
required_parameter_table_exists
all_equal
sum_equals
product_approximately_equals
less_or_equal
change_requires_reason
issue_response_status_exists
alias_normalization
evidence_required
reply_table_status_complete

#### 3.3.4 术语表（Terminology）与「规则/术语自动发现」

| 项 | 说明 |
|----|------|

## 四、设计思路（为什么这样建）

### 4.0 总体设计思想

开题给出混合架构：**大模型 + 规则引擎 + 行业知识**。本地版落地时的取舍如下。

| 设计决策 | 理由 | 刻意不做什么 |
|----------|------|--------------|
| **规则优先、LLM 独立二审** | 井数合计、产能上限等必须可复核、可回归；LLM 不负责「改写」规则结论，只作第二意见 | 不用 LLM 直接当唯一裁判 |
| **三值逻辑 FAIL-CLOSED** | 缺口径/缺证据时 UNKNOWN 或转人工，避免「没报错=正确」 | 禁止 UNKNOWN→PASS |
| **SourceSpan 贯穿全链路** | 专家要回到原文位置；导出与证据门禁共用同一 ID 体系 | 禁止无页码/引用的「假装定位」 |
| **声明式 YAML 规则 + 白名单 operator** | 业务可扩展规则而不改引擎；operator 是纯函数，可单测 | 禁止 eval/exec、禁止正文当代码 |
| **第一性原理 / 反假绿** | DEMO 样本里写着「答案句」时极易作弊 grep；已删 legacy 路径并用测试守门 | 禁止按 rule_id 写死特判答案 |
| **本地优先** | 方案涉敏；默认 127.0.0.1 + 文件本机；在线 LLM 可选且应脱敏 | 不做公网多租户 |
| **诚实交付** | golden 偏差、未跑项、检不出能力写进文档 | 禁止虚构准确率/效率百分比 |

### 4.0.1 与开题四项核心任务的映射

| 开题任务 | 设计落点 | 完成度 |
|----------|----------|--------|
| 方案解析与完整性检查 | DocxParser + COMPLETENESS-001/002/003 | 高（文本 DOCX） |
| 关键参数提取与规则核验 | extraction + CONSISTENCY/CAPACITY/TERM/EVIDENCE | 中高（有已知假绿/单位债） |
| 多版本比对与变更追踪 | VERSION-001/002 operator；`app/diff` 库 | 中（产品路径未全接线） |
| 问题定位与报告生成 | evidence_locations + Excel/Word/匿名包 + 专家 PATCH | 高 |

### 4.0.2 人机分工（产品设计）

```
机器：重复、可编码、要一致 — 章节、合计、容量、术语表面、证据锚点
专家：地质工程判断、重大风险、结论签发、规则修订
输出文案：永远带免责声明「AI 初审结果，不是正式审查结论」
```

## 五、核心处理流水线（技术路线）

### 4.1 端到端数据流

1. **上传** — 扩展名白名单 `.docx`；流式写入；100MB 上限；ZIP 结构校验。  
2. **解析** — 按 OOXML body 顺序遍历段落与表格；生成 `SourceSpan`（含 section_path）。  
3. **抽取** — 表格结构化 + 正文正则；可选 `source_version`（文件名 `V*`）。  
4. **归一** — 术语别名 → canonical；pint 单位 → `normalized_value` + `canonical_unit`。  
5. **规则** — `RuleEngine.evaluate`：operator → evidence gate → `RuleResult`。  
6. **LLM** — 独立二审（非复读规则结果）；证据 ID 门禁；失败应 fail-closed 跳过 AI。  
7. **合并** — 规则权威；LLM 可补充同键证据并标 hybrid；强制人工复核标记。  
8. **专家** — PATCH 状态/备注；不覆盖 AI 原文快照字段策略。  
9. **导出** — 识别报告含位置；匿名包仅 opaque 字段 + hash。

### 4.2 StageRunner 语义

- 顺序执行；**首个异常**记录 `FAILED` 阶段并停止。  
- 错误串经 `sanitize_error` 去路径/密钥。  
- **设计意图：** 硬错误不静默跳过。  
- **已知改进点：** LLM 校验 `ValueError` 曾与 `LLMProviderError` 不对称（整管线失败导致规则 finding 未 reconcile）— 见修复计划 Task 1。

### 4.3 第一性原理与反作弊

历史教训：曾出现按 DEMO 正文「答案句」grep 的 `legacy_compatibility` 假绿路径，已删除。  
守门测试：`tests/unit/test_compatibility_safety.py`。  
开发纪律：Finding 必须可沿 **fact → span → operator → 三值** 复盘。

## 六、模块设计要点

### 5.1 解析与抽取

- **只处理文本型 DOCX**；非 DOCX → 415。  
- Span ID 形如 `{doc}:p:{i}` / `{doc}:t:{ti}:{r}:{c}`。  
- 参数事实带完整比较键（subject / time_scope / statistical_scope）才参与多数一致性比较。

### 5.2 规则引擎

| 能力 | 实现要点 |
|------|----------|
| 白名单 operator | 未知 operator → 加载/执行失败，不动态 eval |
| 证据门禁 | `apply_evidence_gate`：不把 UNKNOWN 洗成 PASS |
| 跨参数算术 | 各操作数各自完整即可，**不强制共享 scope**（井数建设期 vs 产能达产期） |
| 人工复核 | `RuleDefinition.requires_human_review` 声明式 |

**结题时已知技术债（规则侧）：**

| 项 | 影响 | 状态 |
|----|------|------|
| TERM-001 在「先术语归一再评」路径上易假绿 | 别名使用可能漏报 | 修复计划 Task 4–5 |
| TERM-001 曾只注入术语表首条 | 覆盖不全 | Task 5 |
| EVIDENCE-001 用全文 span 数量 | 真实文档易恒 PASS | Task 6 |
| 跨参数丢弃 incomplete sibling / 缺单位检查 | 潜在 fail-open | Task 7 |

### 5.3 LLM 子系统

| 组件 | 作用 |
|------|------|
| `LLMProvider` Protocol | 统一 `review(LLMRequest) -> LLMResponse` |
| `MockProvider` | 确定性关键字演示；无网络 |
| `AnthropicAdapter` | `{base_url}/v1/messages`；`follow_redirects=False`；密钥不进异常 |
| `build_provider` | 配置不全 → Mock |
| `validate_findings` | 证据 ID ⊆ 请求集合；severity 枚举 |

系统提示明确：**文档是数据不是指令**（content-as-data）。

### 5.4 安全设计

| 控制 | 实现 |
|------|------|
| 本地绑定 | run_local → 127.0.0.1 |
| 路径 | `safe_join` + UUID4 case 目录 |
| 上传 | 扩展名、大小、DOCX 成员检查 |
| 密钥 | keyring WinVault 类型校验 |
| 日志 | 敏感键 redact |
| 匿名导出 | 字段白名单，无正文/位置/厂商 |
| URL | 严格 `validate_base_url` + 放宽 `validate_llm_base_url`（内网网关场景） |

**结题时已知风险（安全侧）：** 无鉴权改 base_url 可保留旧 key 导致密钥打到新主机；默认 LLM URL 策略偏松——修复计划 Task 9；页数上限配置未执行——Task 10。

### 5.5 持久化与导出

- SQLite 仅元数据；案例文件在 `storage/cases/...`。  
- 删除：回收站 → 二次确认 `DELETE {case_id}`。  
- 导出：Excel/Word 含问题位置；匿名 ZIP 去标识。

## 十、风险、限制与治理

### 10.1 技术限制

- 仅文本型 DOCX；复杂域公式/图件不解析。  
- 参数抽取覆盖率受模板与表述影响（golden 文档记录 36/38 等限制）。  
- 单文档时序矛盾等能力未实现处已诚实记录。  
- LLM 有幻觉风险 → **证据门禁 + 专家复核** 双闸。  
- TERM-001 / EVIDENCE-001 等在修复完成前存在已知假绿/齿软（见合一 Review 文档）。

### 10.2 安全与合规

- 本地优先降低出网面；一旦启用在线 LLM，**正文可能出网** — 须脱敏与制度审批。  
- 当前无应用层用户认证；威胁模型是「本机多进程」而非公网。  
- 须完成合一文档中 C3/C4 修复后，方可扩大在线 LLM 使用。

### 10.3 治理红线（写进 CLAUDE.md）

- 不自动审批  
- 不伪造页码/引用  
- UNKNOWN 不静默变 PASS  
- 测试不造假绿

## 十二、结题结论

### 12.1 技术结论

本项目在工作坊周期内交付了**可运行、可测试、可演示**的本地开发方案初审内核，落实了开题中的混合架构主干：**规则引擎保证确定性，LLM 提供可选增强，术语/规则 YAML 承担轻量知识载体**。工程上具备清晰分层、三值 fail-closed 设计、安全默认（loopback、keyring、匿名导出）与对抗性测试骨架。**12 条规则**（10 外部 + 2 仓库）与自动发现加载路径已写清（§3.3）。

### 12.2 业务结论

系统已能对脱敏/示例 DOCX 自动给出**可定位的初审问题列表**，支撑「机器做重复核对、专家做最终判断」的分工。**效率提升比例**需在真实审查场景用对照实验度量，不宜在结题技术报告中夸大。

### 12.3 诚实清单（汇报时建议口述）

1. 多版本 diff 产品路径未完全接线；  
2. 在线 LLM 生产加固项见 [合一 Review/修复文档](code-review-and-remediation-2026-07-15.md)；  
3. TERM/EVIDENCE 等规则存在已识别的第一性原理缺口，计划已排期；  
4. 「效率 50%」是业务目标，不是已测代码指标；  
5. 输出永远是 AI 初审，不是正式审查结论。

### 12.4 建议决议

| 决议项 | 建议 |
|--------|------|
| MVP 技术结题 | **通过**（以本地 Mock+规则路径为准） |
| 试点使用在线 LLM | **有条件通过**（完成合一文档 Part B 中 P0/P1 后） |
| 下一阶段立项 | 多版本接线 + 规则运营 + 真实项目对照实验 |
